Personalized Speech Recognition & Transcript Output Dev. time · ~6 months Offline Whisper desktop workflow: transcribe media, edit subtitle-style segments, optional fine-tuning and JA↔EN draft translation.
Repository

Overview

A Windows-first desktop tool chain built around OpenAI Whisper for offline transcription of audio and video. Video is normalized through FFmpeg to 16 kHz mono WAV; the leading segment drives Japanese vs English detection before a full pass. Results land in editable segment lists with export to SubRip (.srt) and JSON, plus optional paths for dataset review, experimental fine-tuning, and draft machine translation for subtitle workflows.

Technologies used

Python 3.9+ PyTorch OpenAI Whisper Tkinter pygame FFmpeg transformers (optional) SRT / JSON

Technical notes

  • Resolving FFmpeg follows a fixed order (next to the frozen executable, PyInstaller _MEIPASS, repo root, then PATH) so packaged and dev runs behave the same.
  • Per-user workspaces under users/<name>/ isolate transcripts, media, and checkpoints without mixing profiles on disk.
  • Fine-tuning is intentionally thin: a small PyTorch loop around Whisper modules suitable for experimentation, not a production adaptation claim—alignment, tokenizer contracts, and evaluation would need hardening for real deployments.
  • Translation uses Hugging Face–style pipelines when enabled, producing draft bilingual subtitles rather than certified translations.

Purpose (why it was built)

To shorten the time spent creating translated subtitles during video editing, and to produce subtitle data aligned with the audio for future speech-recognition model training.

Technical challenges

  • Long end-to-end processing time for transcription and related steps.
  • Fine-tuning involves manual, hands-on work in the workflow (dataset curation and training setup).
Multi-Language Summary API Dev. time · ~2 weeks Self-hosted Flask service: multilingual (JA/EN/ES/ZH) summaries via fast TF-IDF or abstractive mT5, batch API, and local fine-tuning data pipeline.
Repository

Overview

A local-first HTTP API for multilingual news-style summarization in Japanese, English, Spanish, and Chinese. Clients choose between a low-latency extractive TF-IDF path and an abstractive mT5 checkpoint (csebuetnlp/mT5_multilingual_XLSum), with automatic dispatch, graceful fallback when mT5 output is invalid, and rich metadata (engine_used, fallback_used, timing, compression). A batch endpoint summarizes many documents in one round-trip; optional hooks collect (article, summary) pairs for later fine-tuning.

Technologies used

Python Flask waitress / WSGI TF-IDF (custom) Hugging Face mT5 Docker pytest GitHub Actions

Technical notes

  • Language detection combines script heuristics with stop-word statistics (and optional langdetect) so JA / EN / ES / ZH stay disambiguated without a heavy dependency stack.
  • Category-aware TF-IDF uses external JSON keyword lists per language, so editorial tuning does not require core code changes.
  • Operational split: TF-IDF targets sub-100 ms class latency on CPU for indexing-style use; mT5 trades memory and seconds-level latency for abstractive quality—clients pick per request.
  • Fine-tuning loop is documented end-to-end: collect pairs via API, curate JSONL, split, train, then point MT5_MODEL_NAME at the new checkpoint.

Purpose (why it was built)

To support an integrated news-style site by surfacing headlines and short summaries instantly from full articles.

Technical challenges

  • When article length or volume of text is very large, summarization quality can drop compared to shorter inputs.
Security Camera (Android + Azure Functions) Dev. time · ~1 month Kotlin / Compose app with CameraX & ML Kit heuristics, paired with Node.js Azure Functions for embedding-based face matching, Table Storage, and optional ACS email alerts.
Repository

Overview

An end-to-end sample pairing a Kotlin / Jetpack Compose client with Node.js Azure Functions. The app uses CameraX and ML Kit heuristics to capture snapshots, sends base64 JPEGs over HTTPS for embedding-based face matching against templates in Azure Table Storage, and can raise email alerts via Azure Communication Services with optional image attachments. Settings such as the function base URL live in DataStore; the README explicitly calls out that embedding a host key in the app is a demo anti-pattern for production.

Technologies used

Kotlin Jetpack Compose CameraX ML Kit OkHttp DataStore Azure Functions (Node) TensorFlow.js / face-api Azure Table Storage Azure Communication Services

Technical notes

  • Enrollment averages embeddings from multiple JPEGs before upserting a row in Table Storage so a single bad frame is less likely to dominate the template.
  • Monitor mode ties ML Kit person/face heuristics to snapshot timing and server-side cosine similarity, with cooldown logic to avoid alert storms.
  • Backend uses TF.js + face-api style embeddings in Functions; local dev is supported via Azurite and Core Tools per repo docs.
  • Production hardening would replace long-lived function keys in the app with a token gateway, attestation, or short-lived credentials—the sample prioritizes clarity over security theater.

Purpose (why it was built)

To improve security for a room at home, and to gain hands-on experience building an image-processing and face-recognition style application end to end.

Technical challenges

  • Integrating with external triggers (e.g. SwitchBot–class devices) so the phone camera does not need to stay on continuously.
